## Goal

Implement production-ready passkey (WebAuthn) MFA alongside existing TOTP in a Next.js/Supabase Auth/Prisma app, then apply review-driven security hardening.

## Constraints & Preferences

- Keep Supabase Auth as primary auth; no paid auth vendor
- Passkeys are MFA/step-up only, not passwordless primary auth
- Preserve existing password, Google, SSO, break-glass flows
- No placeholder code, no AI slop, no half-finished paths
- Production-ready quality bar

## Review Remediation History

### Round 1 (commit `663d67a`)

- In-memory challenge store replaced with Prisma/Postgres `WebAuthnChallenge` table
- `userVerification: "required"` for registration and authentication
- `getRequiredEnv()` fails closed in production without `WEBAUTHN_RP_ID`/`WEBAUTHN_ORIGIN`
- Password re-auth for passkey removal
- Recovery code fallback generalized for passkey-only users
- Failed challenge audit logging

### Round 2 (commit `bd478e1`)

- **Atomic challenge consumption**: replaced `findMany` + `update` with `findMany` + `updateMany(where: { id, consumed: false })` — parallel consumers get `count: 0` and fail closed
- **Auth-method-agnostic re-auth**: replaced password-only `removePasskey()` with step-up token verification via `src/lib/step-up.ts`
  - `getStepUpFactors()` reads Supabase identities + profile for `{ hasPassword, hasTotp, hasPasskey }`
  - `verifyStepUpPassword()` — `signInWithPassword` for email/password users
  - `verifyStepUpTotp()` — TOTP code verification for authenticator app users
  - `beginStepUpPasskey()` / `verifyStepUpPasskey()` — full WebAuthn ceremony for passkey users
  - `removePasskey()` requires a valid step-up token, not a password
- UI: multi-method step-up dialog in security settings (password input, TOTP code, or passkey browser challenge)

### Round 3 — PR Hygiene (commit `ae7509f`)

- Removed 8 unrelated PRD prompt artifacts from the branch
- PR now scoped only to passkey MFA implementation and its tests

## Key Decisions

| Decision | Rationale |
|----------|-----------|
| Challenge store: Prisma/Postgres over in-memory Map | Multi-replica safety; atomic consumption via `updateMany` guard |
| Step-up tokens (HMAC) over passwords | Works for password, Google, and SSO users without assuming auth provider |
| `userVerification: "required"` for all WebAuthn ceremonies | Production MFA hardening; no silent fallback to "preferred" |
| Recovery codes generalized for MFA | Passkey-only users can still use recovery codes from prior TOTP enrollment |
| `getRequiredEnv()` fail-closed | Production must set `WEBAUTHN_RP_ID`/`WEBAUTHN_ORIGIN` explicitly |
| 5-minute step-up token TTL | Short-lived enough for security, long enough for user interaction |
| Failed challenge auditing: best-effort, non-blocking | Org context may not exist in settings; never block user action |
