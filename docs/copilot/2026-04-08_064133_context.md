## What Is Phase 14?

Phase 14 adds **Accounts Receivable (AR) Automation** to Slipwise One:

1. **Dunning Engine** — Automated payment reminders (5-step sequences, SMS + email)
2. **Customer Portal** — Self-service customer access (JWT-based, zero-code integration)
3. **Quotes** — Convert quotes to invoices, track acceptance
4. **Cash Flow Intelligence** — 90-day forecast, DSO, AR aging, customer health scores
5. **Payment Arrangements** — Installment plans for overdue invoices

**Key architectural choice:** Razorpay only (no Stripe). All payment logic reuses existing services.

## Phase 14 PRD Overview (`docs/PRD/PHASE_14_PRD.md`)

### Document Structure (2,436 lines, 101KB)
1. **Sections 1–21** covering design, schema, routes, APIs, edge cases, tests
2. **Sprint 14.1:** Dunning Engine (5 steps, idempotency, opt-out, auto-renewal)
3. **Sprint 14.2:** Customer Portal (8 pages, JWT tokens, silent refresh)
4. **Sprint 14.3:** Quotes + Cash Flow + Arrangements (state machines, forecasting)

### Key Technical Decisions
- **Dunning idempotency:** Check `DunningLog` before firing any step (never fire twice)
- **SMS provider:** MSG91 (India), DLT-registered, silent skip if `MSG91_API_KEY` missing
- **Portal tokens:** JWT (HS256), 30-day expiry, hashed SHA-256 in DB (never raw)
- **Quotes:** nanoid(16) public tokens, conversion idempotent via `convertedInvoiceId` check
- **Cash flow:** Pre-computed nightly via Trigger.dev job, stored in `ReportSnapshot`
- **Customer health:** 0–100 score, nightly computation, used in Conservative forecast
- **Arrangement safety:** Total must equal remaining amount at creation (prevents debt hiding)

### What Phase 14 Does NOT Include
- Quote versioning (Phase 15)
- Multi-currency or GST e-invoicing (Phase 15)
- Template marketplace or OAuth 2.0 (Phase 15)

## Edge Cases Covered (37 total)

### Dunning (15)
- Invoice paid before step fires → stop dunning
- Step fails → log, skip to next (or retry per config)
- Customer on arrangement → dunning stops
- Multiple DunningSequences configured → use `OrgDefaults.dunningSequenceId`
- Dunning opt-out (HMAC token) → validate scoping, prevent replay
- SMS without `MSG91_API_KEY` → email only
- Payment link expired → auto-renew if <1 day to expiry
- Step 5 creates ticket → check for existing ticket (idempotency)
- Large org (10k+ invoices) → scheduled job batching
- Timezone handling (org.timezone) → interpret `daysOffset` correctly
- Invoice reissued → stop old dunning, start new sequence
- Customer deleted → soft-delete, stop dunning, clean portal tokens
- Org plan downgraded → disable dunning (paid feature)
- User clicks opt-out link → creates DunningOptOut record, stops all future dunning for customer
- Retry mechanism → max 3 retries per step before failing

### Portal (14)
- Token refresh within 15 days remaining → issue new token + Set-Cookie
- Token expired → redirect to login
- Customer deleted → revoke all tokens (invalidate `tokenHash` records)
- Org plan downgraded → disable portal, show upgrade message
- Customer tries to pay someone else's invoice → query scoping prevents (org + customer check)
- Magic link login → never reveal if email exists (security)
- Customer updates email → re-issue portal token with new email
- Concurrent token refresh → use transaction to prevent race
- Portal custom domain → DNS CNAME validation
- Missing `portalCustomDomain` → use default `/portal/[orgSlug]/[token]/`
- Customer has no invoices → show empty state, offer contact support link
- Payment from portal → reuses `createRazorpayPaymentLink()` (no new logic)
- Portal URL share → token in URL path (not query param for security)
- Admin revokes portal access → set `OrgDefaults.enablePortal = false` → token validation fails

### Quotes (10)
- Quote total = 0 → block creation
- Quote to invoice conversion → check `convertedInvoiceId`, skip if already converted
- Edit sent quote → block in Phase 14 (versioning is Phase 15)
- Public quote accept when invoice already exists → link to invoice detail, mark quote ACCEPTED
- Public quote decline → mark DECLINED, send notification
- Quote with no customer → use default email, prompt to update
- Quote expires → no hard delete, but mark expired in UI
- Quote line item with rate = 0 → allow (may be promotional)
- Customer deleted → orphan quote (no danger, customer lookup will fail in portal)
- Quote conversion → create Invoice in same org (validation in service)

### Payment Arrangements (7)
- Arrangement.totalArranged ≠ Invoice.remainingAmount → reject at creation
- Installment paid but invoice paid separately → reconcile correctly
- Arrangement defaulted → revert invoice to OVERDUE, resume dunning
- Installment marked WAIVED → don't count toward amountPaid in reconciliation
- Zero-amount installment → allow (admin waiver scenario)
- Arrangement with one installment = original invoice → block (use invoice due date instead)
- Org deletes arrangement → orphan becomes historical record

## Key Assumptions & Decisions

1. **No Stripe.** Razorpay only. All payment logic reuses existing services.
2. **India-first.** MSG91 SMS, Razorpay, GST e-invoicing Phase 15.
3. **JWT portal tokens** — stateless, 30-day expiry, refreshable within 15 days.
4. **Dunning idempotency** — check `DunningLog` before firing; never fire step twice.
5. **Org context scoping** — every portal query filters by `orgId + customerId` (strict multi-tenancy).
6. **Reuse payment services** — `reconcileInvoicePayment()`, `createRazorpayPaymentLink()` never rewritten.
7. **Plan gating** — dunning (Free+), portal (Starter+), quotes (Starter+), cash flow (Pro+).
8. **Manual migration** — cannot use `prisma migrate dev` due to prior schema drift.
