## Passkey MFA Feature Context

Goal: add production-grade passkey MFA alongside the existing auth system without adding a new paid auth provider.

Current auth model:

- Supabase Auth remains the primary auth/session layer.
- Existing primary sign-in methods stay intact:
  - email/password
  - Google
  - enterprise SSO
  - break-glass flows
- Existing TOTP and recovery-code support stays intact.
- Passkeys are implemented as MFA/step-up, not as primary passwordless auth.

Main implementation in PR `#196`:

- Added `@simplewebauthn/server` and `@simplewebauthn/browser`.
- Added Prisma `PasskeyCredential` model for WebAuthn credentials.
- Added Prisma `WebAuthnChallenge` model for durable challenge storage.
- Added passkey server helpers in `src/lib/passkey/server.ts`.
- Added durable challenge store in `src/lib/passkey/challenge-store.ts`.
- Added passkey DB helpers in `src/lib/passkey/db.ts`.
- Added browser passkey helpers in `src/lib/passkey/client.ts`.
- Added step-up token helper in `src/lib/step-up.ts`.
- Added passkey enrollment, listing, rename, removal, and step-up server actions in `src/app/app/settings/security/passkey-actions.ts`.
- Updated `/auth/2fa` to become a multi-method MFA challenge surface with passkey, TOTP, and recovery-code paths.
- Updated security settings UI to manage passkeys and preserve TOTP/recovery fallback.
- Updated middleware to use generalized MFA semantics through user metadata and the existing MFA completion cookie.
- Added audit events for passkey add/rename/remove/use/failure and related MFA events.

## Passkey Review History

Initial review findings on PR `#196`:

- In-memory WebAuthn challenge store was not production-safe.
- WebAuthn user verification was set to `preferred`/`false`, too weak for MFA.
- Passkey removal only used a browser confirm and lacked re-authentication.
- Failed passkey challenges were not audited.
- Recovery codes were still effectively TOTP-only.
- Unrelated `docs/PRD/*` prompt artifacts were included in the PR.

First remediation completed:

- Replaced in-memory challenges with Prisma-backed `WebAuthnChallenge`.
- Required WebAuthn user verification.
- Added fail-closed production env guards for RP/origin config.
- Added password re-auth for passkey removal.
- Generalized recovery-code fallback.
- Added failed passkey challenge audit logging.

Second review findings:

- Durable challenge store used read-then-update, leaving a concurrency race.
- Password-only passkey removal re-auth broke Google/SSO-only users.

Second remediation completed:

- `consumeChallenge()` and `getAndConsumeChallenge()` now use atomic `updateMany({ id, consumed: false })` claims.
- Passkey removal now requires a short-lived signed step-up token.
- Step-up can be satisfied by password, TOTP, or passkey.
- UI offers available step-up methods dynamically.
- Audit log for passkey removal includes the step-up method.

Final PR hygiene remediation:

- Removed 8 unrelated `docs/PRD/*` prompt artifacts from the PR.
- Final branch diff is scoped to passkey/auth code, Prisma migrations, middleware, and tests.

Final review call:
