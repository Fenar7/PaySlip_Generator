1. Document Purpose
This document defines the complete product transformation of Slipwise into Slipwise One, a multi-product business document and workflow suite.
This PRD is intended to be the master handoff document for:
* product planning
* engineering architecture
* sprint planning
* design direction
* QA planning
* future hiring and delivery sequencing
This is not a lightweight idea note. It is the operational blueprint for taking the current product from a stateless document generator into a scalable SaaS suite.
The current Slipwise product already has live document modules for Voucher, Salary Slip, Invoice, and PDF Studio, runs on Vercel, uses Next.js 16 + React 19 + TypeScript + Tailwind CSS, and intentionally excludes accounts, databases, storage, and recurring workflows. PDF Studio is already quite advanced and includes HEIC/HEIF, OCR, watermarking, page numbers, AES-256 password protection, compression, and metadata controls.
The workshop notes clearly expand the ambition of the product toward onboarding, persistent design defaults, stored data, invoice statuses, payment proof, dashboards, sending, scheduling, recurring invoices, multi-user access, proxy controls, full PDF utility coverage, and passport photo editing.
This document restructures all of that into one coherent suite.

The workshop notes move the product beyond static generation and into persistent business workflows. The requested additions are not small. They include:
* Zoho-style direct-input design
* template store
* multi-line vouchers
* invoice query/ticketing
* client-side payment status marking
* partial payment support

* cloud storage and business data retention

* proxy actions with audit history

This means the product is no longer one app with a few extra features.
It is now a suite architecture problem.

6. Strategic Product Principles
6.1 Persist the workflow, not just the file
A PDF alone is not the product.
The product is:
   * saved record
   * state
   * history
   * proof
   * reminder
   * auditability
   * team actionability
6.2 Keep the paid core separate from the free utility layer
PDF Studio and SW Pixel are growth levers.
They should not derail the core SaaS.
6.3 Build for 1,000 to 10,000 users without re-architecture
The product should scale cleanly to that range with modular monolith architecture.
6.4 Avoid premature complexity
Do not start with microservices, event-driven overengineering, or infrastructure cosplay.
6.5 Build suite clarity, not feature sprawl
Every feature must clearly belong to one suite module, even if it interacts with others.

10.1.3 Invoice Studio
Current Baseline
The current invoice generator already supports:
   * business/client blocks
   * shipping support
   * tax IDs
   * place of supply
   * dynamic line items
   * tax and discount math
   * amount in words
   * notes, terms, bank, signature, payment summary
   * template switching
   * export PDF/PNG/print
Future Scope
Invoice Studio will support:
   * persistent save/edit/reopen
   * customer linking
   * template assignment
   * default workspace styles
   * invoice status lifecycle
   * original vs reissue
   * reissue reason
   * payment state
   * client query/ticket initiation
   * send and resend
   * scheduled send
   * recurring generation trigger
   * document timeline
   * attachment support
   * invoice vault listing
   * filters and search
Business Rules
   * invoice number rules configurable by workspace
   * draft and issued are separate states
   * once paid or partially paid, state changes are recorded, not overwritten
   * reissue must preserve audit trail
   * invoice templates must be overrideable per document
   * invoice can be linked to customer, project, or department later
Acceptance Criteria
   * invoice can be created, saved, resumed, duplicated, archived
   * lifecycle state is visible
   * payment state is stored
   * send logs are visible
   * linked client ticket and proof uploads are accessible
________________


10.1.4 Voucher Studio
Current Baseline
Current voucher module supports payment/receipt vouchers, branding, optional fields, amount in words, live preview, templates, and export.
Future Scope
Voucher Studio will support:
   * persistent records
   * multi-line voucher mode
   * date and time per line
   * grouped day or multi-day voucher
   * approval state
   * voucher category summaries
   * receipt and payment subtype logic
   * audit trail
   * template store support
   * operator-specific access
Business Problem Solved
The workshop note explicitly highlights the hospitality use case where multiple petty cash events in a day should not force multiple separate vouchers. A multi-line voucher structure solves that operational pain.
Acceptance Criteria
   * user can create single-line or multi-line voucher
   * grouped entries calculate totals correctly
   * voucher can be filtered by date/user/category
   * voucher can be printed/exported in clean grouped layout
________________
